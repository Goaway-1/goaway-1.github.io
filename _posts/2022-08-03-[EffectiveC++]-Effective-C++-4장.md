## 들어가며

인터페이스를 설계할 때 가장 중요한 지침은 <span style = "color:orange;">__"제대로 쓰기엔 쉽게, 엉터리로 쓰기엔 어렵게"__</span> 이다.

## 항목 18 : 인터페이스 설게는 제대로 쓰기엔 쉽게, 엉터리로 쓰기엔 어렵게 하자

"제대로 쓰기엔 쉽게, 엉터리로 쓰기엔 어렵게"를 만족하는 인터페이스를 개발하기 위해서는 사용자가 저지를 만한 실수의 종류를 상기하고 있어야 한다. 가장 쉬운 방법은 __int나 STL과 같이 이미 존재하는 인터페이스의 동작을 그대로 본뜨는 것이다.__ 그렇게 되면 일관성을 유지할 수 있기에 사용자 쪽에서 뭔가를 외워야할 필요가 없다.

> 1. 매개변수의 전달 순서가 잘못된 경우

  ```cpp
  class Date{
    Date(int month, int day, int year);
  }
  
  //ERROR
  Date d(30, 3, 2022);   //30일을 30달로 잘못 사용
  ```

  위와 같이 날짜를 나타내는 클래스가 있다고 가정했을때, 문제는 없어보이지만 사용자가 사용함에 있어 매개변수의 전달 순서가 잘못되는 경우가 발생할 수 있다.

  ```cpp
  struct Month{
    explict Month(int d) : val(d) {}
    int val;
  }
  ...
  class Date{
    Date(const Month& m, const Day& d, const Year& y);
  }

  // 해결
  Date d(Month(3),Day(30),Year(2022));
  ```

  이를 해결해주는 것이 바로 타입 시스템이며, 예시에서는 랩퍼(Wrapper) 타입을 만들어 해결한다. Wrapper란 기본 자료형같은 데이터를 객체에 넣기 위해 제공하는 함수들이라고 생각하면 된다.

> 2. 타입의 범위를 벗어나는 경우

  ```cpp
  struct Month{
    static Month Jan() {return Month(1);}
    static Month Feb() {return Month(2);}
    ...
  }

  // 해결
  Date d(Month::Mar(),Day(30),Year(2022));
  ```

  매개변수의 값이 허용범위 외의 값이 들어올 수 있다. 이를 위해서는 enum을 사용하는 방법이 ㅇ존재하지만 이는 타입의 안전성이 낮기 때문에, 유효한 집합을 미리 정의해두어 사용할 수 있다. (함수)

> 3. 스마트 포인터의 사용을 잊는 경우

  ```cpp
  Investment *createInvestment();

  // 사용자가 스마트 포인터의 사용을 잊는 경우 해결
  tr1::shared_ptr<Investment> createInvestment();
  ```

  위 createInvestment()를 사용하는 경우에는 자원 누출을 피하기 위해서 포인터를 나중에라도 삭제해야 한다. 하지만 사용자가 포인터의 삭제를 잊거나 똑같은 포인터에 대해 두 번 이상 적용될 수 있다. 

  이를 해결하기 위해 이전 항목 13에서 반환 값을 포인터에 저장한후에 해당 포인터의 삭제 작업을 스마트 포인터에게 떠넘기는 방법을 사용했다. 하지만 이 스마트 포인터를 사용해야 한다는 사실 조차 잊게 된다면 의미가 없게된다. 그렇기에 __애초부터 스마트 포인터를 반환하게 만듦으로서 해결한다.__

  이렇게 하면 함수의 반환 값은 tr1::shared_ptr에 넣어둘 수밖에 없고, 나중에 Investment 객체가 필요 없을 경우 삭제하는 것을 잊어도 상관없다. <span style = "color:orange;">__사실 tr1::shared_ptr을 반환하는 구조는 자원 해제에 관련된 상당수의 사용자 실수를 사전 처리할 수 있기에 인터페이스 설계자에게 유용하게 사용된다.__</span">

  ```cpp
  // 다른 함수에게 포인터 삭제 인계
  tr1::shared_ptr<Investment> pInv(0,getRidOfInvestment);                             //컴파일 안됌
  tr1::shared_ptr<Investment> pInv(static_cast<Investment*>0,getRidOfInvestment);     //컴파일 가능
  ```

  추가로 포인터를 직접 삭제하지 않고, 다른 함수(getRidOfInvestment)에게 넘기는 방법이 존재하는데, 이렇게 하면 자원 해제 매커니즘을 잘못 사용할 수가 없도록 한다. 
  
  이러한 __"사용자 정의 삭제자"__ 덕분에 tr1::shared_ptr에는 가장 좋은 특징이 하나 있는데, 포인터별 삭제자를 자동으로 사용하므로써 __'교차 DLL 문제'를__ 방지한다. 이는 객체 생성 시에 동적 링크 라이브러리의 new를 썼는데, 그 객체를 삭제할 때 이전의 DLL과 다른 DLL에 있는 delete를 쓰는 경우 런타임 에러가 발생하는 것을 의미한다. 하지만 tr1::shared_ptr은 다른 DLL들 사이에 이리저리 넘겨지더라도 어떤 DLL의 delete를 사용해야 하는지 알고 있기 때문에 교차 DLL 문제를 걱정할 필요가 없다.

### 잊지 말자

  - 좋은 인터페이스는 __"제대로 쓰기에 쉬우며 엉터리로 쓰기 어렵다."__ 인터페이스를 만들때는 이 특성을 지닐 수 있도록 구현하자.
  - 인터페이스의 올바른 사용을 이끄는 방법으로는 인터페이스 사이의 일관성 잡아주기, 그리고 기본제공 타입과의 동작 호환성 유지하기가 있다.
  - 사용자의 실수를 방지하는 방법으로는 새로운 타입 만들기, 타입에 대한 연산을 제한하기, 객체의 값에 대해 제약 걸기, 자원 관리 작업을 사용자 책임으로 놓지 않기가 있다.
  - tr1::shared_ptr은 사용자 정의 삭제자를 지원한다. 이 특징 때문에 tr1::shared_ptr은 교차 DLL 문제를 막아주며, 뮤텍스 등을 자동으로 잠금 해제하는 데 쓸 수 있다.
