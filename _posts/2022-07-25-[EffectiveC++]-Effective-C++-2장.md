## 항목 5 : c++가 은근슬쩍 만들어 호출해 버리는 함수들에 촉각을 세우자

기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자는 직접 개발자가 선언해 놓지 않으면, 컴파일러가 스스로 기본형으로 선언한다.

이때 예를들어 Object 클래스 내에, 멤버 변수가 각각 __참조자, 상수로__ 선언되어있고, 개발자가 복사 생성자를 선언하지 않아 컴파일러가 기본형으로 선언되었다면 오류가 발생한다.

```cpp
//header
template <class T>
class Object{
private:
  string &nameValue;
  const T objectValue;
}

//cpp
Object<int> p(a,2);
Object<int> s(b,4);
p = s;    //ERROR
```

그 이유는 c++의 참조자는 원래 자신이 참조하고 있는 것과 다른 객체를 참조할 수 없기 때문이다. 또한 상수 객체의 경우도 마찬가지이다.

그렇기 때문에 참조자나 상수가 있는 클래스의 경우에는 __함수를__ 컴파일러에게 의존하지 않고, 개발자가 직접 선언해두는 것이 좋다.

### 잊지 말자

  - 컴파일러는 경우에 따라 클래스에 대해 기본 생성자, 복사 생성자, 복사 대입 연산자, 소멸자를 암시적으로 만들어 놓을 수 있다.

## 항목 6 : 컴파일러가 만들어낸 함수가 필요 없으면 사용을 금해 버리자

__항목 5에서__ 컴파일러가 자동으로 만들어낸 함수에 대해서 사용을 금지하고 싶다면 딱 2가지만 진행하면 된다.

1. 별도의 기본 클래스에 금지하고 싶은 기본 생성 함수를 public 멤버가 아닌 private 멤버로 전환
2. 기본 클래스로 부터 새로운 클래스를 파생시킨다.

즉, A라는 클래스가 있을때 "복사 생성자"의 사용을 금지하고 싶다면, 별도의 기본 클래스에 "복사 생성자"를 private의 멤버로 넣어두고 구현은 하지 않는다. 그리고 이 기본 클래스에서 A라는 클래스를 파생시키는 것이다.

이때 기본 클래스는 복사의 방지만을 맡는다는 특별한 의미를 가지게 된다. 이렇게 되면 __A 객체의 복사를 외부함수나 프렌드 함수에서 시도할때 호출하지 못하도록 막는다.__

### 잊지 말자

  - 컴파일러에서 자동으로 제공하는 기능을 허용치 않으려면, 대응되는 멤버 함수를 private으로 선언한 후에 구현은 하지 않은 채로 둔다.

## 항목 7 : 다형성을 가진 기본 클래스에서는 소멸자를 반드시 가상 소멸자로 선언하자

만약 객체에 대한 포인터를 반환하는 용도로 __팩토리 함수를__ 만든다고 가정해보자

> 팩토리 함수 : 새로 파생된 클래스 객체에 대해 기본 클래스 포인터를 반환하는 함수

```c++
Class A{
  A();
  ~A():

  A* GetA();    //팩토리 함수
}

class B : public A {};
class C : public A {};
```

이때 팩토리 함수의 규약을 따라간다면, GetA()에서 반환되는 객체는 힙에 있기 때문에 메모리 누출을 막기 위해서 해당 객체를 적절히 삭제해야한다. 하지만 이때 3가지의 문제가 발생하게 된다.

1. 이때 반환되는 포인터가 파생 클래스 객체에 대한 포인터
2. 포인터가 가르키는 객체 삭제 시 기본 클래스 포인터를 통해 삭제
3. 기본 클래스의 소멸자가 비가상 소멸자

C++은 기본 클래스 포인터를 통해 파생 클래스의 객체가 삭제될 시 기본 클래스에 __비가상 소멸자가__ 있다면 동작하지 않는다. 즉, 기본 클래스만 삭제되는 "부분 소멸"이 이루어진다는 것을 의미한다.

이를 해결하기 위한 방법은 단순히 __기본 클래스의 비가상 소멸자를 가상 소멸자로__ 바꾸는 것이다.

```c++
Class A{
  A();
  virtual ~A():
}
```
### 가상 소멸자의 남용

> 가상 함수를 가진 클래스가 아닌데 가상 소멸자를 사용하는 경우

가상 소멸자를 항상 사용해서는 안된다. 조건에 맞게 사용해야하는데 위와 같이 가상 함수가 없는데 가상 소멸자를 사용하면 효율성이 떨어지는데 예시는 아래와 같다.

만약 64비트의 자료가 있는 클래스의 소멸자가 가상 소멸자라면, 가상이기 때문에 실행 중에 어떤 가상함수를 호출해야하는지 정보가 필요하기 때문에 추가로 자료구조가 들어간다. 이는 vptr, vtbl로 각각 가상 함수 테이블 포인터, 가상 함수 테이블을 의미한다. 이 때문에 클래스가 64비트 레지스터에 들어가지 못하고, 다른 언어로 선언된 자료구조와의 호환성에도 영향을 미친다.

그렇기 때문에 가상 소멸자는 클래스에 가상함수가 1개 이상 존재하는 경우에서 사용해야한다.

### 순수 가상 소멸자

> 어떤 클래스를 추상 클래스로 만들고 싶은데, 마땅히 순수 가상 함수를 넣을 곳이 없는 경우

```cpp
class A{
  virtual ~A() = 0; //선언
}

A::~A() {}    //정의
```

위와 같이 단순히 순수 가상 소멸자를 선언하고, 그에 대한 정의를 해준다. 

### 잊지 말자

 - 다형성을 갖는 기본 클래스에는 반드시 가상 소멸자를 선언해야한다.
 - 기본 클래스로 설계되지 않았거나 다형을 갖지 않는 클래스에는 가상 소멸자를 선언하지 말아야한다.

## 항목 8 : 예외가 소멸자를 떠나지 못하도록 붙들어 놓자

> 소멸자로부터 예외가 발생하는 경우는 개발자가 막아야 한다. 

만약 예외를 던지고 실패할 수 있는 코드를 소멸자에 넣어야된다고 가정하자. 아래와 같은 경우 블록의 범위를 벗어나면 자동으로 소멸자(~B)가 호출되는데, 이때 예외가 발생했다면, 아무런 처리를 하지 않는다.

```cpp
class A{
  void close();
}
class B{
  ~B{A.close();}    //실패할 수 있음
}
```

이를 해결하기 위해서는 쉽게 2가지 방법이 존재한다.

1. 프로그램을 종료한다.
    ```cpp
    B::~B(){
      try{A.close();}
      catch(...){std::abort();}//강제 종료
    }
    ```

2. 예외를 삼켜 버린다.
    ```cpp
    B::~B(){
      try{A.close();}
      catch(...){ log } //로그만 띄움
    }
    ```
    - 이는 정보만 알려주며, 지속적으로 프로그램이 진행되기 위해서는 신뢰성 있게 실행을 지속할 수 있어야한다.

위 두가지 방법은 각각 장단점이 있지만, 딱히 좋지는 않다. __*대신 사용자가 컨트롤을 하는 방법이 있다.*__

이는 close()메서드를 사용자가 직접 호출하도록하고, 소멸자에서는 사용자의 컨트롤에 따라 위험이 발생할 수 있는 close()메서드를 실행하거나 실행하지 않도록 한다.
그렇게 되면 소멸자가 아닌 다른 함수에서 예외처리가 비롯되기 때문에 __사용자에게 기회를 주고 처리하는 것이다.__

```cpp
class B{
  void close(){
    A.close();
    closed = true;
  }
  ~B(){
    if(!closed){
      try {A.close();}
      catch(...) {//로그 또는 종료}
    }
  }
}
```

### 잊지 말자

 - 소멸자에서는 예외가 빠져나가면 안된다. 만약 그럴 가능성이 있다면 소멸자에서 모두 받아내고, 끝내거나 삼켜 버려야한다.
 - 어떤 클래스의 연산이 진행되다가 예외에 대해 사용자가 반응해야 할 필요가 있다면, 해당 연산을 제공하는 함수는 반드시 소멸자가 아닌 함수이어야 한다.

## 항목 9 : 객체 생성 및 소멸 과정 중에는 절대로 가상 함수를 호출하지 말자

```cpp
class A{
  A();
  virtual void getA();
}
A::A(){ getA(); }
class B :: public A {~~};

//ERROR
B b;    //생성자에 가상함수를 호출하기에
```

위와 같은 경우 기본 클래스의 생성자가 호출될 때에 가상함수는 파생 클래스쪽으로 내려가지 않기 때문에 컴파일러가 에러를 뿜을 것이다. 이렇듯 어떤 객체의 초기화되지 않은 영역을 건드리는 것은 위험하며, 생성자 뿐만 아니라 소멸자도 포함된다.

또 다른 중요한 점은 파생 클래스 객체의 기본 클래스 부분이 생성되는 동안은, 그 객체의 타입은 바로 __기본 클래스라는 점이다.__ 

이를 해결하기 위한 대처방법은 바로, "가상함수를 비가상 함수로 바꾸고, 파생 클래스의 생성자들로 하여금 필요한 로그 정보를 기본 클래스의 생성자로 넘여야 한다는 것이다."

```cpp
class A{
  explicit A(const string& logInfo);
  void logA(const string& logInfo) const;
}
A::A(const string& logInfo){ logA(logInfo); }

class B :: public A{
  // 로그 정보를 기본 클래스 생성자로 넘김
  B(parameters) : A(createLogString(parameters)) {~~}

  // 정적멤버 : 기본 클래스 생성자로 넘길값을 생성하는 용도
  static string createLogString(parameters);
}
```

위 예시처럼 필요한 초기화 정보를 파생 클래스 쪽에서 기본 클래스 생성자로 "올려"주도록 설정하여 부족한 부분을 역으로 채울 수 있다는 것이다. 여기서 createlogString()메서드는 기본 클래스 생성 쪽으로 넘길 값을 생성한느 용도로 쓰이는 도우미 함수이다. 단순히 안쓰는게 짱인듯 하다.

### 잊지 말자

  - 생성자/소멸자 안에서 가상 함수를 호출하지 마세요. 가상 함수여도 클래스의 파생 클래스 쪽으로는 내려가지 않는다.

## 항목 10 : 대입 연산자는 *this의 참조자를 반환하자

C++에서 대입 연산은 여러 개가 사슬처럼 엮일 수도 있고, 우측 연관 연산이다. 여기서 우측 연관이란 오른쪽의 값이 모두 왼쪽으로 대입된다는 것이다.

그렇기 때문에 대입 연산자가 좌변 인자에 대한 참조자를 반환하도록 구현해야한다. 이미 구현되어 있지만 사용자가 직접 만드는 클래스에 대해서는 이 관례를 지켜야한다.

```cpp
class A{
  A & operator=(const A& rhs) {return *this;}
  A & operator+=(const A& rhs) {return *this;}
}
```

### 잊지 말자

  - 대입 연산자는 *this의 참조자를 반환하도록 만들자.

## 항목 11 : operator =에서는 자기대입에 대한 처리가 빠지지 않도록 하자

자기 대입 : 객체가 자기 자신에 대해 대입 연산자를 적용하는 것

```cpp
//자기 대입의 예시
a = a;
a[i] = a[j];
*pa = *pb;      //같은 객체를 참조하는 경우..
```

여러 곳에서 하나의 객체를 참조하려는 상태, 즉 "중복참조"때문에 자기대입이 발생한다. 이 자기 대입에 대해 발생하지 않도록 처리해야한다.

```cpp
class A{~~};
class B{
  A *a;
}
B::operator=(const B& rs){
  delete a;
  a = new A(*rs.a);
  return *this
}
```

위는 자기 참조에 대한 가능성이 존재하는 예시이다. 만약 연산자의 대상이 같은 객체라면 delete 연산자가 *this 객체의 비트맵에만 적용되는 것이 아니라 rs의 객체까지 적용되어 버린다. 그렇기 때문에 기존 포인터 객체가 삭제될 수 있다.

이를 해결하기 위한 방법에는 3가지가 존재한다.

> 1. 일치성 검사를 통해 자기대입을 점검

```cpp
B::operator=(const B& rs){
  if(this == &rs) return *this; //일치성 검사
  delete a;
  a = new A(*rs.a);
  return *this
}
```

두 객체가 같은지 비교하여 자기대입의 경우라면 그냥 왼쪽 객체를 반환하는 방식이다. 이는 예외가 발생할 수 있는데 바로 "new A()"이다. __만약 이 부분에서 예외 발생(메모리 부족, 복사 생성자 예외) 시 B객체는 삭제된 A를 가리키는 포인터를 가지고 남게 된다.__ 

> 2. 무작정 삭제하지 말고, 포인터가 가르키는 객체를 복사한 직후에 삭제

```cpp
B::operator=(const B& rs){
  A *pa = a;          //원래 객체 저장
  a = new A(*rs.a);
  delete pa;
  return *this
}
```

이는 'new A'에서 예외가 발생하더라도 a는 변경되지 않은 상태가 유지되기 때문에 예외에서 안전하다. 가장 효과적인 방법은 아니나 동작에는 이상이 없다.

> 3. 복사 후 맞바꾸기

```cpp
class B{
  void swap(B& rs); //*this와 rs의 데이터를 교환
}
B::operator=(const B& rs){   
  B tmp(rs);
  swap(tmp);
  return *this
}
```

이는 가장 자주 쓰이는 방식으로 추후에 더 자세히 설명된다.

### 잊지 말자

  - operator=를 구현할때, 어떤 객체가 자신에 대입되는 경우를 제대로 처리하도록 만들자. 원본과 복사 대상의 객체 주소를 비교해도 되고, 문장의 순서를 적절히 조정, 복사 후 맞바꾸기 기법을 사용해도된다.
  - 두 개 이상의 객체에 대해 동작하는 함수가 있다면, 넘겨지는 객체들이 사실 같은 객체인 경우에 정확하게 동작하는지 확인하자.

## 항목 12 : 객체의 모든 부분을 빠짐없이 복사하자

복사 함수에는 복사 생성자와 복사 대입 연산자로 나뉘는데, 이 복사 함수는 컴파일러가 자동으로 생성하기도 하지만 사용자가 선언 및 정의하는 경우가 있다. 컴파일러가 자동으로 만들어주는 복사 함수는 객체가 가지고 있는 데이터를 빠짐없이 복사하지만 사용자의 경우 놓칠 수 있다.

딱 2가지만 생각하며 구상하면 되는데, 먼저 클래스의 모든 멤버를 처리한다는 것과 파생클래스의 복사 함수에서 기본 클래스의 복사 함수를 호출하면 된다.

> 1. 클래스의 모든 멤버를 처리하자.

이는 정말 단순하게 클래스의 모든 멤버를 처리하면 된다는 뜻이고, 새롭게 데이터 멤버를 추가했다면 추가한 데이터 멤버 또한 처리되도록 복사 함수를 다시 작성해야한다는 것이다.

> 2. 파생 클래스의 복사 함수에서 기본 클래스의 복사 함수를 호출하자.

파생 클래스의 복사 함수에서는 파생 클래스의 데이터 멤버함수에만 접근할 수 있기 때문에(대부분 기본 클래스의 데이터는 private), 파생 클래스의 복사 함수에서는 복사가 불가능하다. 그렇기 때문에 기본 클래스의 복사 함수를 호출하기만 하면된다. 

```cpp
B::operator=(~~){
  A::operator=(~~);
  ~~~
}
```

### 잊지 말자

  - 객체 복사 함수는 주어진 객체의 모든 데이터 멤버 및 모든 기본 클래스 부분을 빠뜨리지 말고 복사해야 한다.
  - 클래스의 복사 함수 두 개를 구현할 때, 한쪽을 이용해서 다른 쪽을 구현하려는 시도는 하지말라. 대신 공통된 동작을 제3의 함수에다 분리해 놓고 양쪽에서 이것을 호출하게 만들어서 해결합니다.
