#게임#게임 이론

게임 엔진 정의

게임 제작에 필요한 기본 체계와 데이터를 종합적으로 관리해주는 소프트웨어로 개발 과정을 단축시켜주고, 다양한 플랫폼에서 실행할 수 있도록 도와준다. 가상 세계를 만드는 ‘수학’과 콘텐츠를 구성하는 ‘데이터’를 효과적으로 관리한다.

구성요소

게임 엔진은 씬과 리소스로 구성되어 있으며, 이 두가지를 결합하여 렌더링을 한다.

씬에는 게임 오브젝트(트랜스톰, 메시/텍스처 리소스 키)로 구성되고, 리소스에는 메시/텍스처 리소스(리소스 키, 데이터)로 구성된다.

모델링 행렬의 설계

게임 오브젝트의 배치 정보를 “위치, 회전, 크기”로 구성된 프랜스폼의 정보를 사용해 관리하며, 아래와 같다.

데이터 구성 아핀 변환행렬
크기 (S) 2차원 벡터 ($s_x$, $s_y$) \[\begin{bmatrix} s_x &amp; 0 &amp; 0 \ 0 &amp; s_y &amp; 0 \ 0 &amp; 0 &amp; 1 \end{bmatrix}\]
회전 (R) 각 각도 ($\partial$) \[\begin{bmatrix} \cos\partial &amp; -\sin\partial &amp; 0 \ \sin\partial &amp; \cos\partial &amp; 0 \ 0 &amp; 0 &amp; 1 \end{bmatrix}\]
이동 (T) 2차원 벡터 ($t_x$, $t_y$) \[\begin{bmatrix} 1 &amp; 0 &amp; t_x \ 0 &amp; 1 &amp; t_y \ 0 &amp; 0 &amp; 1 \end{bmatrix}\]

트랜스폼의 변환은 “크기 -> 회전 -> 위치”순서로 이뤄지며, 행렬 곱으로 정리하면 “$T \cdot R \cdot S $”이고, 모델링 행렬(M)은 아래와 같다.

$$\begin{bmatrix} \cos\partial\cdot s_x & -\sin\partial\cdot s_y & t_x \\ \sin\partial\cdot s_x & \cos\partial\cdot s_y & t_y \\ 0 & 0 & 1 \end{bmatrix}$$

로컬 공간과 로컬 축

게임에는 오브젝트마다 ‘월드 공간’과 ‘로컬 공간’이라는 두 공간 개념이 존재하며, 여러 물체가 모여 배경을 이루는 것이 월드 공간이고, 독립적인 단일 메시 정보만 표시하는 공간은 로컬 공간이다.

축 이름
X Right Vector
Y Up Vector
Z Forward Vector

리소스 관리

리소스 데이터는 여러 오브젝트들이 함께 사용할 수 있는 공유 자원의 형태로 관리하여 메모리 공간을 효율적으로 사용하며, 이 리소스를 모아두는 저장소가 ‘리소스 저장소’ 라고 하고, 아래와 같은 특징을 지닌다.

  1. 리소스를 유형별로 나누고 각 리소스에 고유한 키를 부여
  2. 게임 오브젝트에는 리소스 데이터를 가리키는 리소스 키 값만 저장하도록 설계하여, 메모리 공간의 소비를 줄임

워크플로우

게임 엔진의 실행 흐름을 “워크플로우”라고 하며, 씬을 완성하는 과정과 완성된 씬으로부터 화면을 그려내는 과정 2가지로 구성된다.

워크 플로우

  1. 리소스 로딩 : 메시/텍스처 리소스를 진행 중에 불러들이기에는 데이터 양이 크기 때문에, 시작 전에 불러들여 메모리에 미리 올려둠
  2. 씬 구축 : 오브젝트의 트랜스폼 정보와 리소스 정보가 설정
  3. 게임 로직 : 프레임마다 트랜스폼 값을 변경
  4. 렌더링 로직 : 트랜스폼을 중심으로 메시/텍스처의 리소스 데이터를 활용해 최종 화면을 그림 (렌더링 파이프라인으로 세분화)

렌더링 파이프라인

GPU 내부에 설정된 워크플로우를 “렌더링 파이프라인”이라 하며, 아래 순서와 같다.

  1. 정점 변환 : 오브젝트에 연결된 메시의 모든 정점의 정보를 변환 (월드 공간을 중심으로 지정)
  2. 정점 처리 : 정점의 최종 위치 값을 구하는 과정으로 모델링 행렬(M)에 로컬 공간의 좌표를 곱하여 월드 공간 기준 좌표로 변환
  3. 픽셀화 : 메시를 구성하는 삼각형마다 픽셀화. 무게중심좌표를 사용하여 색상 결정
  4. 픽셀 처리 : 색상을 가져오거나 조명 적용하여 각 픽셀의 최종 색상을 지정.

루프

게임 루프는 게임을 업데이트하고, 렌더링하는 것입니다. 이 루프는 하드웨어에 따라 속도에 차이가 발생할 수도 있기 때문에 타이밍을 맞추기 위해서는 프레임을 건너뛰어야합니다.

※ FPS (Frame Per Second) : 1초에 몇 프레임을 처리할 수 있는지 ※ Game Speed : 초당 업데이트되는 횟수 (Update()함수가 1초에 호출되는 횟수)

이 FPS를 설정하는 방법에는 ‘고정 FPS(25프레임)’, ‘가변 FPS’, ‘최대 FPS’, ‘가변 FPS와 일정한 게임 속도’등등 다양한 방법이 있지만, 그 중 가장 많이 사용하는 것만 설명하겠습니다.

  • 가변 FPS에 따른 게임속도 
      while (running) {
    prevFrameMs = currFrameMs;
    currFrameMs = GetMilliseconds();

    /** 이전 프레임과의 시간차이를 계산 */
    const uint32 deltaMs = currFrameMs - prevFrameMs;
    Update(deltaMs);
    Render();
  }
    • 루프를 빠르게 실행하고, FPS와 게임 속도가 같도록 하는 것입니다. 이 게임은 이전 프레임의 시간차이로 업데이트됩니다.
    • 대부분의 엔진이 이러한 방식으로 사용되고 있다.
    • 느린 하드웨어 : 많은 오브젝트가 있는 경우 프레임의 저하로 모든 시스템 무너짐
    • 빠른 하드웨어 : 정밀도로 인한 유실이 발생할 수 있다. (float..)

더 많은 예시와 자세한 사항은 참고자료를 확인..

카메라 시스템

월드 공간의 일부분을 랜더링하기 위해서 카메라를 중심으로 물체의 트랜스폼을 재조정하는 작업을 통해 변환된 공간을 “뷰 공간”이라고 한다.

이전 모델링 행렬(M)과 유사하게 월드 좌표를 카메라 중심의 좌표로 변환하는 행렬을 “뷰 행렬(V)” 이라고 하며, 아래와 같다.

$$\begin{bmatrix} 1 & 0 & -t_x \\ 0 & 1 & -t_y \\ 0 & 0 & 1 \end{bmatrix}$$

즉, 카메라를 중심으로 재배치하는 수식은 아래와 같다.

$$ v_v (view) = V \cdot M \cdot v_l (local) $$
