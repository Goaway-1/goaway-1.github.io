코딩테스트 | 캐시

queue, unordered_map

Jul 24, 2022 About 6 mins

#CodingTest#C++

문제

지도개발팀에서 근무하는 제이지는 지도에서 도시 이름을 검색하면 해당 도시와 관련된 맛집 게시물들을 데이터베이스에서 읽어 보여주는 서비스를 개발하고 있다.

이 프로그램의 테스팅 업무를 담당하고 있는 어피치는 서비스를 오픈하기 전 각 로직에 대한 성능 측정을 수행하였는데, 제이지가 작성한 부분 중 데이터베이스에서 게시물을 가져오는 부분의 실행시간이 너무 오래 걸린다는 것을 알게 되었다.

어피치는 제이지에게 해당 로직을 개선하라고 닦달하기 시작하였고, 제이지는 DB 캐시를 적용하여 성능 개선을 시도하고 있지만 캐시 크기를 얼마로 해야 효율적인지 몰라 난감한 상황이다.

어피치에게 시달리는 제이지를 도와, DB 캐시를 적용할 때 캐시 크기에 따른 실행시간 측정 프로그램을 작성하시오.

조건

  • 캐시 크기(cacheSize)와 도시이름 배열(cities)을 입력받는다.
  • cacheSize는 정수이며, 범위는 0 ≦ cacheSize ≦ 30 이다.
  • cities는 도시 이름으로 이뤄진 문자열 배열로, 최대 도시 수는 100,000개이다.
  • 각 도시 이름은 공백, 숫자, 특수문자 등이 없는 영문자로 구성되며, 대소문자 구분을 하지 않는다. 도시 이름은 최대 20자로 이루어져 있다.
  • 캐시 교체 알고리즘은 LRU(Least Recently Used)를 사용한다.
  • cache hit일 경우 실행시간은 1이다.
  • cache miss일 경우 실행시간은 5이다.

입출력 예시

캐시크기(cacheSize) 도시이름(cities) 실행시간
3 [“Jeju”, “Pangyo”, “Seoul”, “NewYork”, “LA”, “Jeju”, “Pangyo”, “Seoul”, “NewYork”, “LA”] 50
3 [“Jeju”, “Pangyo”, “Seoul”, “Jeju”, “Pangyo”, “Seoul”, “Jeju”, “Pangyo”, “Seoul”] 21
2 [“Jeju”, “Pangyo”, “Seoul”, “NewYork”, “LA”, “SanFrancisco”, “Seoul”, “Rome”, “Paris”, “Jeju”, “NewYork”, “Rome”] 60
0 [“Jeju”, “Pangyo”, “Seoul”, “NewYork”, “LA”] 25

풀이방식

나의 풀이방식

먼저 도시의 이름은 대소문자를 구분하지 않는다는 조건이 있었기 때문에 도시의 이름을 toBigger()라는 메서드를 사용하여, 모두 대문자로 처리해주었다.

문제의 해결 방식은 단순하다. map에는 현재 queue에 저장된 도시가 무엇이 있는지 나타내며, 이 queue는 cache를 의미한다.

  1. cache에 있는 도시라면 그 도시를 제외한 나머지를 다시 cache에 넣고, 그 도시를 마지막으로 삽입한다. (+1)
  2. 그렇지 않다면 cache의 공간이 남아있는지 확인한다. - 남아 있다면 단순히 cache에 추가하고 (+5) - 공간이 없다면 cache의 최상단(front)를 제거하고 추가한다. (+5)
  #include <algorithm>
  #include <queue>
  #include <unordered_map>

  using namespace std;

  string toBigger(const string& str) {
    string answer = "";
    for (int i = 0; i < str.size(); i++){
      answer += toupper(str[i]);
    }
    return answer;
  }

  int solution(int cacheSize, vector<string> cities) {
    int answer = 0;
    unordered_map<string, int> M;
    queue<string> Q;

    if (cacheSize == 0) return cities.size() * 5;
    for (int i = 0; i < cities.size(); i++){
      string tmp = toBigger(cities[i]);
      
      // Q에 있는 단어라면 queue의 가장 뒤로 뺀다.
      if (M[tmp] != 0) {
        queue<string> tmpQ;
        while (!Q.empty()) {
          if (Q.front() != tmp) tmpQ.push(Q.front());
          Q.pop();
        }
        tmpQ.push(tmp);
        Q = tmpQ;
        answer += 1;
      }
      else {
        // Q의 크기를 넘어간다면 최상단을 제거한다.
        if (Q.size() == cacheSize) {
          M[Q.front()] = 0;
          Q.pop();
        }

        M[tmp]++;
        Q.push(tmp);
        answer += 5;
      }
    }
    return answer;
  }

다른이의 풀이방식

비슷한 방식이지만 map을 사용하지 않고, 매번 iterator를 사용하여 비교하였다.

또한 queue 대신 vector를 사용하였고, 중간에 껴있는 도시에 대해 “erase”를 하기 위해서 사용한듯 하다.

  #include <algorithm>

  using namespace std;
  int solution(int cacheSize, vector<string> cities) {

    vector <string> q;
    int duration = 0;

    for(vector <string>::iterator it = cities.begin(); it != cities.end(); it++){
      //소문자화
      transform(it->begin(), it->end(), it->begin(), ::tolower);

      //현재 Q에 있는지 판단
      bool flag = false;
      for(vector<string>::iterator itt = q.begin(); itt != q.end(); itt++){
        if(*itt == *it) {
          flag = true;
          duration +=1;
          q.erase(itt);
          q.push_back(*it);
          break;
        }
      }
      if(!flag) {
        duration +=5;
        if(cacheSize != 0 && q.size() >= cacheSize) q.erase(q.begin());
        if(q.size() < cacheSize) q.push_back(*it);
      }
    }

    return duration;
  }

개선점

  • map을 사용하였기 때문에 메모리적인 측면에서 비효율적이지 않나 생각이 든다.
