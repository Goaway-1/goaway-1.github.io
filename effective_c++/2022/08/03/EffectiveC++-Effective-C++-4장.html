항목 18 : 인터페이스 설게는 제대로 쓰기엔 쉽게, 엉터리로 쓰기엔 어렵게 하자

“제대로 쓰기엔 쉽게, 엉터리로 쓰기엔 어렵게”를 만족하는 인터페이스를 개발하기 위해서는 사용자가 저지를 만한 실수의 종류를 상기하고 있어야 한다. 가장 쉬운 방법은 int나 STL과 같이 이미 존재하는 인터페이스의 동작을 그대로 본뜨는 것이다. 그렇게 되면 일관성을 유지할 수 있기에 사용자 쪽에서 뭔가를 외워야할 필요가 없다.

  1. 매개변수의 전달 순서가 잘못된 경우
  class Date{
    Date(int month, int day, int year);
  }
  
  //ERROR
  Date d(30, 3, 2022);   //30일을 30달로 잘못 사용

위와 같이 날짜를 나타내는 클래스가 있다고 가정했을때, 문제는 없어보이지만 사용자가 사용함에 있어 매개변수의 전달 순서가 잘못되는 경우가 발생할 수 있다.

  struct Month{
    explict Month(int d) : val(d) {}
    int val;
  }
  ...
  class Date{
    Date(const Month& m, const Day& d, const Year& y);
  }

  // 해결
  Date d(Month(3),Day(30),Year(2022));

이를 해결해주는 것이 바로 타입 시스템이며, 예시에서는 랩퍼(Wrapper) 타입을 만들어 해결한다. Wrapper란 기본 자료형같은 데이터를 객체에 넣기 위해 제공하는 함수들이라고 생각하면 된다.

  1. 타입의 범위를 벗어나는 경우
  struct Month{
    static Month Jan() {return Month(1);}
    static Month Feb() {return Month(2);}
    ...
  }

  // 해결
  Date d(Month::Mar(),Day(30),Year(2022));

매개변수의 값이 허용범위 외의 값이 들어올 수 있다. 이를 위해서는 enum을 사용하는 방법이 ㅇ존재하지만 이는 타입의 안전성이 낮기 때문에, 유효한 집합을 미리 정의해두어 사용할 수 있다. (함수)

  1. 스마트 포인터의 사용을 잊는 경우
  Investment *createInvestment();

  // 사용자가 스마트 포인터의 사용을 잊는 경우 해결
  tr1::shared_ptr<Investment> createInvestment();

위 createInvestment()를 사용하는 경우에는 자원 누출을 피하기 위해서 포인터를 나중에라도 삭제해야 한다. 하지만 사용자가 포인터의 삭제를 잊거나 똑같은 포인터에 대해 두 번 이상 적용될 수 있다.

이를 해결하기 위해 이전 항목 13에서 반환 값을 포인터에 저장한후에 해당 포인터의 삭제 작업을 스마트 포인터에게 떠넘기는 방법을 사용했다. 하지만 이 스마트 포인터를 사용해야 한다는 사실 조차 잊게 된다면 의미가 없게된다. 그렇기에 애초부터 스마트 포인터를 반환하게 만듦으로서 해결한다.

이렇게 하면 함수의 반환 값은 tr1::shared_ptr에 넣어둘 수밖에 없고, 나중에 Investment 객체가 필요 없을 경우 삭제하는 것을 잊어도 상관없다. 사실 tr1::shared_ptr을 반환하는 구조는 자원 해제에 관련된 상당수의 사용자 실수를 사전 처리할 수 있기에 인터페이스 설계자에게 유용하게 사용된다.</span”>

  // 다른 함수에게 포인터 삭제 인계
  tr1::shared_ptr<Investment> pInv(0,getRidOfInvestment);                             //컴파일 안됌
  tr1::shared_ptr<Investment> pInv(static_cast<Investment*>0,getRidOfInvestment);     //컴파일 가능

추가로 포인터를 직접 삭제하지 않고, 다른 함수(getRidOfInvestment)에게 넘기는 방법이 존재하는데, 이렇게 하면 자원 해제 매커니즘을 잘못 사용할 수가 없도록 한다.

이러한 “사용자 정의 삭제자” 덕분에 tr1::shared_ptr에는 가장 좋은 특징이 하나 있는데, 포인터별 삭제자를 자동으로 사용하므로써 ‘교차 DLL 문제’를 방지한다. 이는 객체 생성 시에 동적 링크 라이브러리의 new를 썼는데, 그 객체를 삭제할 때 이전의 DLL과 다른 DLL에 있는 delete를 쓰는 경우 런타임 에러가 발생하는 것을 의미한다. 하지만 tr1::shared_ptr은 다른 DLL들 사이에 이리저리 넘겨지더라도 어떤 DLL의 delete를 사용해야 하는지 알고 있기 때문에 교차 DLL 문제를 걱정할 필요가 없다.

잊지 말자

  • 좋은 인터페이스는 “제대로 쓰기에 쉬우며 엉터리로 쓰기 어렵다.” 인터페이스를 만들때는 이 특성을 지닐 수 있도록 구현하자.
  • 인터페이스의 올바른 사용을 이끄는 방법으로는 인터페이스 사이의 일관성 잡아주기, 그리고 기본제공 타입과의 동작 호환성 유지하기가 있다.
  • 사용자의 실수를 방지하는 방법으로는 새로운 타입 만들기, 타입에 대한 연산을 제한하기, 객체의 값에 대해 제약 걸기, 자원 관리 작업을 사용자 책임으로 놓지 않기가 있다.
  • tr1::shared_ptr은 사용자 정의 삭제자를 지원한다. 이 특징 때문에 tr1::shared_ptr은 교차 DLL 문제를 막아주며, 뮤텍스 등을 자동으로 잠금 해제하는 데 쓸 수 있다.

항목 19 : 클래스 설게는 타입 설게와 똑같이 취급하자

좋은 클래스를 설게하기 위해서는 문법(syntax)이 자연스럽고, 의미구조(semantics)가 직관적이며, 효율적인 구현이 한 가지 이상 가능해야한다. 클래스를 설계할 때 도움이 될만한 질문을 아래에 제시한다.

새로 정의한 타입의 객체 생성 및 소멸은 어떻게 이루어져야 하는가?

  • 이 부분에 따라 클래스 생성자/소멸자의 설계가 바뀌며, 메모리 할당함수를 직접 작성할 경우에는 함수의 설계에도 영향을 미친다. (항목 8,16)

객체 초기화는 객체 대입과 어떻게 달라야 하는가?

  • 생성자와 대입 연산자의 동작의 차이점을 결정지으며, 초기화와 대입을 헷갈리면 안된다. (항목 4)

새로운 타입으로 만든 객체가 값에 의해 전달되는 경우에 어떤 의미를 줄 것인가?

  • ‘값에 의한 전달’을 구현하는 쪽은 복사 생성자이다.

새로운 타입이 가질 수 있는 적법한 값에 대한 제약은 무엇으로 잡을 것인가?

  • 데이터 멤버의 몇 가지 조합 값만은 반드시 유효해야하며, 이 조합을 클래스의 불변속성이라고 한다. 생성자/대입 연산자/쓰기 함수는 불변속성에 좌우된다.

기존의 클래스 상속 계통망에 맞출 것인가?

  • 기존 클래스로부터 상속시킨다면 제약을 받게되며, 특히 멤버 함수가 가상/비가상인지 여부에 따라 큰 영향을 미친다. 특히 소멸자를 구성할때 조심해라. (항목 7)

어떤 종류의 타입 변환을 허용할 것인가?

  • 생성한 타입과 다른 타입 사이에 변환수단이 필요할때, 암시적으로 변환하려면 ‘변환 함수’를 하나 넣어두고, 명시적으로 변환하려면 ‘별도 이름의 함수’를 만들어 반환한다. (항목 15)

표준 함수들 중 어떤 것을 허용하지 말 것인가?

  • private으로 선언해서 기본제공함수를 사용하지 못하게 막는다. (항목 6)

‘선언되지 않은 인터페이스’로 무엇을 둘 것인가?

  • 만든 타입이 제공할 보장이 어떤 종류일까에 대한 질문으로, 보장할 수 있는 부분은 수행 성능/예외 안전성/자원 사용이다. 이 보장은 클래스 구현에 있어 제약으로 작동한다.

새로 만드는 타입이 얼마나 일반적인가?

  • 정의하는 것이 동일 계열의 타입군 전체일지도 모른다. 그럼 새로운 것이 아니다.

정말로 꼭 필요한 타입인가?

  • 기존 클래스에 대해 기능이 아쉬워서 파생 클래스를 새로 만들고 있다면, 차라리 비멤버 함수라든지 템플릿을 몇 개 더 정의하는 편이 낫다.

어떤 연산자와 함수를 두어야 의미가 있을까?

새로운 타입의 멤버에 대한 접근권한을 어느 쪽에 줄 것인가?

잊지 말자

  • 클래스 설게는 타입 설계이다. 새로운 타입을 정의하기 전에 이번 항목에 나온 모든 고려사항을 빠짐없이 점검해 보십시오.

항목 20 : ‘값에 의한 전달’보다는 ‘상수객체 참조자에 의한 전달’ 방식을 택하는 편이 대개 낫다.

기본적으로 C++는 함수로부터 객체를 전달받거나 함수에 객체를 전달할 때, “값에 의한 전달”방식을 사용한다. (C의 특성)

class Person{
  // 생성자, 소멸자 생략
private:
  string name;
  string address;
};
class Student : public Person{
  // 생성자, 소멸자 생략
private:
  string schoolName;
  string schooladdress; 
}

// 값에 의한 전달
bool A(Student s);
Student plato;
bool isA = A(plato);

위와 같이 함수 A에 대해서 값에 의한 절달하는 방식은 문제없이 정상적으로 작동한다. 하지만 여기서 문제는 “복사손실”이 발생한다는 점이다. 이는 의미없는 복사가 발생하여 비용에 있어 손해를 보는 것이다. 위는 생성자/소멸자가 각각 6번 호출된다.

// 상수객체에 대한 참조자
bool A(const Student &s);

이렇게 복사손실을 막기 위한 방법이 “상수객체에 대한 참조자”를 사용하는 것이다. 이렇게 하면 새로 만들어지는 객체가 없기 때문에 생성자와 소멸자가 호출되지 않기에 복사손실 문제가 발생하지 않는다.

class Window{
  string name() const;
  virtual void display() const;
};
class windowWithScrollBars : public Window{
  virtual void display() const;
};

// 복사손실 + windowWithScrollBars의 display는 절대 호출되지 않음
void printNameAndDisplay(Window w){
  cout << w.name();
  w.display();
}

위의 예시를 보자. window 클래스가 있고 이를 상속받는 windowWithScrollBars 클래스가 있다고 가정했을때 printNameAndDisplay()메서드에서의 매개변수는 Window이다. 이는 w에 어떤 타입의 객체가 넘겨지든 아랑곳없이 Window 클래스 객체의 면모만을 가지게 되는 문제가 발생한다. 즉, 가상함수 WindowWithScrollBar::display()메서드는 절대로 호출되지 않는다.

// 문제 발생하지 않음
void printNameAndDisplay(const Window &w){
  cout << w.name();
  w.display();
}

하지만 ‘상수객체 참조자에 의한 전달’을 사용하면 w에 어떤 타입의 객체가 넘겨지더라도 그 윈도우의 성질을 그대로 가지게 된다.

이렇듯 ‘상수객체 참조자에 의한 전달’을 사용하면 다양한 메모리적, 타입별 문제가 발생하는 것을 막아준다. 그렇다고 ‘값에 의한 전달’을 사용하지 말라는 뜻은 아니다. 저비용으로 가정해도 괜찮은 유일한 타입은 기본제공 타입, STL 반복자, 함수 객체 타입으로 이때 ‘값에 의한 전달’을 사용해도 괜찮다. 그 외의 타입에 대해서는 ‘상수객체 참조자에 의한 전달’을 선택하는 것이 올바른 방법이다.

잊지 말자

  • ‘값에 의한 전달’보다는 ‘상수 객체 참조자에 의한 전달’을 선호합시다. 대체적으로 효율적일뿐만 아니라 복사손실 문제까지 막아 줍니다.
  • 이번 항목에서 다룬 법칙은 기본제공 타입 및 STL 반복자, 그리고 함수 객체 타입에는 맞지 않습니다. 이들에 대해서는 ‘값에 의한 전달’이 더 적절합니다.

//반환이 참조자 
const Rational& operator *(const Rational& lhs, const Rational& rhs){

위 같이 참조자를 반환한다면 비용 부담은 확실히 없을 것이다. 하지만 참조자는 존재하는 객체에 붙는 다른 이름이다. 그렇기 때문에 참조자를 반환하기 위해서는 반드시 존재하는 Ratioal의 객체이여야 한다. 만약 위처럼 3가지 방법이 있다고 .

1번은 result에 대한 참조자를 반환하는데, 이 result는 다른 객체처럼 생성되어야 하고 지역객체이기 때문에 함수가 끝날 때 소멸된다. 그렇기 때문에 아무런 의미 없는 바이트 덩어리가 반환된다.

2번은 new로 생성된 객체인데 이 또한 다른 객체처럼 생성되어야 하고, 누가 delete로 뒤처리를 해줄것인가..?라는 큰 문제가 발생한다.

3번은 정적 객체이기 때문에 모든 객체에 값이 공유가 된다. 즉 원하는 결과를 도출할 수 었다.

이처럼 참조자를 반환하는 것은 많은 문제를 야기하고, 해결에 있어 어려움이 존재한다.

inline const Rational operator*(const Rational& lhs, const Rational& rhs){
  return Rational(lhs.n * rhs.n,lhs.d * rhs.d)
}

그냥 새로운 객체를 반환하게 만드는 것, 즉 값을 반환하면 쉽게 해결할 수 있다. 이 과정에 있어 위 1,2,3번 문제처럼 생성자와 소멸자가 호출되지만 C++에서도 컴파일러 구현자들이 가시적인 동작 변경을 가하지 않고도 기존 코드의 수행성능을 높이는 최적화를 적용할 수 있도록 배려해 두었다. 그 결과 몇몇 조건하에서는 이 최적화 매커니즘에 의해 operator*의 반환 값에 대한 생성과 소멸 동작이 안전하게 제거될 수 있다.

잊지 말자

  • 지역 스택 객체에 대한 포인터나 참조자를 반환하는 일, 혹은 힙에 할당된 객체에 대한 참조자를 반화하는 일, 또는 지역 정적 객체에 대한 포인터나 참조자를 반환하는 일은 그런 객체가 두 개 이상 필요해질 가능성이 있다면 절대로 하지 마세요.

항목 22 : 데이터 멤버가 선언될 곳은 private 영역임을 명심하자

데이터 멤버가 private 멤버이어야 한다는 이유는 아래 3가지를 통해서 설명할 수 있다.

문법적 일관성

  • 어떤 클래스의 공개 인터페이스에 있는 것들이 전부 함수뿐이라면, 그 클래스의 멤버에 접근하고 싶을 대 괄호를 붙여야 하는지 말아야 하는지를 기억하지 못해서 사용자가 헷갈라는 경우가 없을 것이다. 단순히 함수만 사용하면된다.

정교한 제어

  • 어떤 데이터 멤버가 public이라면 이 멤버에 대해 읽기/쓰기 접근권한을 가지게 되지만, private으로 선언하고 멤버를 읽기/쓰기하는 함수가 있다면 권한을 직접 구현할 수 있다.
  • “접근 불가/읽기 전용/읽기 쓰기 가능”을 말이다.

캡슐화

  • 함수를 통해서만 데이터 멤버에 접근할 수 있도록 구현해 두면, 데이터 멤버를 나중에 계산식으로 대체할 수도 있을 것이고, 사용자는 절대로 이 클래스를 넘볼 수 없게 된다.
  • 클래스의 불변속성을 항상 유지하는데 신경쓸 수 밖에 없다.
  • 데이터 멤버를 함수 인터페이스 뒤에 감추면 융통성 또한 누릴 수 있다.
    • 데이터 멤버를 읽거나 쓸 대 다른 객체에 알림 메시지를 보낸다.
    • 클래스의 불변속성 및 사전/사후조건을 검증
    • 스레딩 환경에서 동기화를 건다.

이때 protected는 어떠냐고 생각할 수 있지만, 이 또한 public과 별반 다를것이 없다. 만약 protected데이터 멤버를 제거한다고 가정했을 때, 이 멤버에 의존하는 다른 코드를 해결하기 위해서 힘을 써야할 것이다. 그니까 그냥 데이터 멤버는 private을 쓰자.

잊지 말자

  • 데이터 멤버는 private 멤버로 선언합시다. 이를 통해 클래스 제작자는 문법적으로 일관성이 있는 데이터 접근 통로를 제공할 수 있고, 필요에 따라서는 세밀한 접근 제어도 가능하며, 클래스의 불변속성을 강화할 수 있을 뿐 아니라, 내부 구현의 융통성도 발휘할 수 있습니다.
  • protected는 public보다 더 많이 보호받고 있는 것이 절대로 아니다.

항목 23 : 멤버 함수보다는 비멤버 비프렌드 함수와 더 가까워지자

class WebBrowser{
public:
  void clearCache();
  void clearHistory();
  void removeCookies();
  
  // 1. 멤버함수로의 구현 (위 3가지 함수 모두 호출)
  void clearEveryThing();   
};

// 2. 비멤버 비프렌드 함수로의 구현
void clearBrowser(WebBrowser& wb){
  wb.clearCache;
  wb.clearHistory;
  wb.removeCookies;
}

위와 같이 WebBrower클래스의 멤버 함수들을 한번에 처리하는 함수를 구현하고자 할때, 1.멤버함수(clearEveryThing)로 구현하거나 2.비멤버 비프렌드 함수(clearBrower)로 구현할 수 있다.

결론부터 말하면 “비멤버 비프렌드 함수”로 구현한 clearBrower()가 훨씬 좋다. 객체 지향 법칙에 따라 멤버함수로 한 곳에 묶어야 된다는 것은 잘못된 생각이다. “비멤버 비프렌드 함수”로 구현하면 캡슐화 정도에서 효과적이며, 컴파일 의존도는 낮추고 패키지 유연성과 확장성도 높일 수 있다.

캡슐화 측면에서의 효율

캡슐화 측명에 있어 비멤버 비프렌드 함수가 어떤 효율성을 지니는지에 대해 설명하고자 한다. 캡슐화를 하게되면 그만큼 밖에서 볼 수 있는 데이터가 줄어들고, 그 것을 수정하기 위해서는 필요한 유연성이 커지게 된다. 그렇기 때문에 캡슐화가 늘어나면, 변경할 수 있는 여유도 많아진다. 즉, 이미 있는 코드를 바꾸더라도 제한된 사용자들밖에 영향을 주지 않는 융통성을 확보할 수 있다.

또한 멤버 함수는 private 멤버뿐 아니라 모든 데이터에 접근이 가능하지만, 비멤버 비프렌드 함수는 어느 것에도 접근할 수 없고, 오직 public에만 접근이 가능하다. 이는 어떤 클래스의 private 멤버 부분을 접근할 수 있는 함수의 개수를 늘리지 않기 때문에 캡슐화의 정도를 높여준다.

※ 이때 주의할 부분은 두 가지가 존재한다. 첫째, 비멤버 비프렌드 함수에만 적용된다는 점이다. 프렌드 함수는 private 멤버에 대한 접근권한이 멤버함수와 동일하기 때문에, 캡슐화에 대한 영향 역시 같다. 둘째, “함수는 어떤 클래스의 비멤버가 되어야 한다”는 이번 항목의 주장이 “그 함수는 다른 클래스의 멤버가 될 수 없다”는 의미는 아니다. 단순히 정적 멤버 함수로 만들어도 되지만, 클래스의 멤버가 아니기만 하면된다.

네임스페이스 안에 두는 방법

  namespace WebBrowerStuff{
    class WebBrowser {...};
    void clearBrowser(WebBroswer& wb);
  }

보다 더 자연스러운 방법으로 구사하기 위해서는 clearBrowser를 비멤버 함수로 두되, WebBrowerStuff와 같은 네임스페이스 안에 두는 것이다. 네임스페이스는 클래스와 달리 여러 개의 소스 파일에 나뉘어 흩어질 수 있는 장점이 있는데, clearBrowser와 같은 함수는 편의상 준비한 함수이기 때문이다.

  //webbrowser.h
  namespace WebBrowerStuff{
    class WebBrowser {...};
    // 핵심 기능만 구현. 즉,거의 모든 사용자가 써야하는 비멤버 함수들이 여기에 들어감
  }

  //webbrowserBookmarks.h
  namespace WebBrowerStuff{
    // 즐겨찾기와 관련된 편의함수(비멤버 함수)들이 여기에 들어감
  }

  //webbrowsercookies.h
  namespace WebBrowerStuff{
    // 쿠키와 관련된 편의함수(비멤버 함수)들이 여기에 들어감
  }

이렇게 되면 컴파일에 대한 의존성을 고민할 이유가 없다. 이를 쉽게 나누는 방법은 기능/사용 범위에 따라 헤더 파일에 따로 선언하는 것이다. 위와 같이 즐겨찾기/쿠키와 관련된 편의함수는 다른 헤더에 작성해두고, 모든 사용자가 사용할 편의함수(비멤버 비프렌드 함수)는 기본 네임스페이스에 선언한다.

즉, 먼저 편의 함수를 하나의 헤더 파일에 몰아서 선언하고, 관련된 장르에 따라 다른 헤더 파일에 각각 몰아서 선언하는 것이다.

잊지 말자

  • 멤버 함수보다는 비멤버 비프렌드 함수를 자주 쓰도록 합시다. 캡슐화 정도가 높아지고, 패키징 유연성도 커지며, 기능적인 확장성도 늘어납니다.

항목 24 : 타입 변환이 모든 매개변수에 대해 적용되어야 한다면 비멤버 함수를 선언하자

“암시적 타입 변환을 지원하는 것은 많은 오류를 초래한다.”라는 것에 예외가 있는데, 바로 숫자 타입을 만들 때이다. 예를 들어 유리수를 나타내는 클래스를 만들고 있다면, 정수에서 유리수로의 암시적 변환은 허용해도 되기 때문이다.

class Rational{
public:
  Rational(int num = 0, int den = 1);   //explicit를 선언하지 않고, 암시적 변환을 가능하도록..
  int num() const;
  int den() const;

  // 1. 멤버 함수로의 제작
  const Rational operator*(const Rational& rhs) const;
}

유리수를 나타내는 클래스이니 덧셈이나 곱셈 등의 수치 연산을 기본으로 제공하고자 한다. 이때 멤버/비멤버 중 어떻게 선언해야할지 고민이 될 것이다.

//1번 멤버 함수로의 제작
Rational A(1,2);
Rational B(3,4);

Rational C = A * B;   //GOOD
C = A * 2;            //GOOD   -> A.operator*(2);
C = 2 * A;            //ERROR  -> 2.operator*(A);

먼저 1번 멤버 함수로 제작했을 경우이다. “C = A * 2”에서 A 객체는 operator* 함수를 멤버로 가지고 있기 때문에, 컴파일러는 이 함수를 호출하기 때문에 잘 구동된다. 이때 숫자 2는 컴파일러가 Rational을 요구한다는 사실을 알고 있기 때문에 숫자 2를 암시적 타입 변환으로 “Rational(2)”로 둔갑한다. 하지만 마지막 “C = 2 * A”는 성립되지 않는다. 그 이유는 “2.operator*(A)”의 형태를 지니고 있기 때문인데, 이는 정수 2에서 operator* 함수를 가지고 있지 않기 때문이다.

이때 operator함수에 명시호출(explicit)로 선언하게되면 암시적 타입 변환이 이루어지지 않기 때문에 위 예시는 모두 오류가 발생한다.

위 예시로 알 수 있는 사실은 “암시적 타입 변환에 대해 매개변수가 먹혀들려면 매개변수 리스트에 들어가야한다는 점”이다. 즉, 호출되는 멤버 함수를 갖고 있는(this) 객체에 해당하는 암시적 매개변수에는 암시적 변환이 먹히지 않는다. operator*를 비멤버 함수로 만들어서, 컴파일러 쪽에서 모든 인자에 대해 암시적 타입 변환을 수행하도록 한다면 될 것이다.

//2번 비멤버 함수로의 제작
class Rational{ ... };
const Rational operator*(const Rational &lhs, const Rational &rhs) { ... };

// TEST
Rational C = A * B;   //GOOD
C = A * 2;            //GOOD   
C = 2 * A;            //GOOD

이처럼 operator*를 비멤버 함수로 만들어서, 컴파일러 쪽에서 모든 인자에 대해 암시적 타입 변환을 수행하도록 구현하였다. 두 문장의 동작이 일관되게 동작하고, 혼합형 수치 연산도 올바르게 작동하는 것을 확인할 수 있다.

프렌드 함수로 만들어도 되지 않나? 라고 생각할 수 는 있지만, 프렌드는 private에도 접근이 가능하기 때문에 별로 추천하지 않는다. 즉, “멤버 함수의 반대는 프렌드 함수가 아닌 비멤버 함수”이다. 프렌드 함수는 되도록이면 피하자.

나의 생각

멤버함수로 구현하고, this에 대해서 매개변수를 구현하면 되지 않나? 그래서 해봤다.

안된다. 비멤버함수로 구현하면 2개의 매개변수를 넣을 수 있지만, 멤버함수로 구현하면 1개의 매개변수밖에 안된다. 왜냐하면 operator에 A라는 객체가 이미 포함되어 있기 때문이다.

잊지 말자

  • 어떤 함수에 들어가는 모든 매개변수(this 포인터가 가르키는 객체도 포함해서)에 대해 타입 변환을 해 줄 필요가 있다면, 그 함수는 비멤버이어야 한다.

항목 25 : 예외를 던지지 않는 swap에 대한 지원도 생각해보자

// swap의 원형 : std내에 존재
namespace std{
  template<Typename T>
  void swap(T &a, T &b){
    T temp(a);
    a = b;
    b = temp;
  }
}

swap은 초장기부터 STL에 포함된 이래로 예외 안전성 프로그래밍에 없어선 안 될 감초 역할로서, 자기대입 현상의 가능성에 대처하기 위한 대표적인 메커니즘으로서 널리 사용되었다. 이는 두 객체의 값을 “맞바꾸기”하여, 각자의 값을 상대방에게 주는 동작을 한다. 위는 swap의 원형인데, 한 번 호출 시 복사가 세 번발생한다. 만약 데이터가 포인터일때 swap을 사용하면 큰 손해를 볼 수 있다.

클래스의 경우

// pimpl 관용구기법
class WidgetImpl{   //실제 데이터가 담긴 클래스
private:
  int a,b,c;
  vector<double> v;
};
class Widget{
public:
  Widget(const Widget &rhs);
  Widget& operator = (const Widget &rhs){
    ...
    *pImpl = *(rhs.pImpl);
  }
  void swap(Widget &other) {
    using std::swap;
    swap(pImpl, other.pImpl);
  }
private:
  WidgetImpl *pImpl;    //Widget의 실제 데이터를 가지는 포인터
}

//수정된 swap
namespace std{
  template<>
  void swap<Widget>(Widget &a, Widget &b){
    a.swap(b);
  }
}

이를 해결하기 위한 기법이 바로 “pimpl 관용구”이다. 이는 pImpl 포인터만 살짝 바꾸면 swap을 사용할 수 있다. 이 정보를 알리기 위해서 “std::swap”을 위와 같이 수정한다. 함수 이름 뒤에 을 붙여 템플릿 특수화 함수라는 것을 컴파일러에게 알려준다. 일반적으로 std 구성요소는 변경이 불가능하나, 프로그래머가 직접 만든 타입에 대해 표준 템플릿을 완전 특수화하는 것은 허용이 된다.

즉, 두 객체의 값을 빠르게 바꿀 함수 swap을 public 멤버로 만들고, 특수화된 std::swap에서 이를 호출하도록 만든다.

클래스 템플릿의 경우

//클래스 템플릿의 경우
template<typename T>
class WidgetImpl {...};

// 비멤버 swap
namespace WidgetStuff{
  template<typename T>
  class Widget {...};   //swap이란 멤버 함수 존재

  template<typename T>  // 비멤버 swap함수
  void swap(Widget<T> &a, Widget<T> &b){
    a.swap(b);
  }
}

만약 클래스가 아닌 클래스 템플릿으로 구성된 경우에는 기존과 동일하게 구현하면 안된다. C++는 클래스 템플릿에 대해서 부분 특수화를 허용하지만 함수 템플릿에 대해서는 허용하지 않기 때문이다. 그렇기 때문에 이를 위해서는 “멤버 swap을 호출하는 비멤버 swap을 선언해 놓되, 이 비멤버 함수를 std::swap의 특수화 버전, 오버로딩 버전으로 선언하지만 않으면 된다.”

이제는 어떤 코드가 swap을 호출하더라도, 컴파일러는 c++의 이름 탐색 규칙(인자 기반 탐색, 쾨니그 탐색)에 의해 WidgetStuff 네임스페이스 안에서 widget 특수화 버전을 찾아낸다. 하지만 “클래스 타입 전용의 swap”이 호출되도록 하고 싶다면, 그 클래스와 동일한 네임스페이스 안에 비멤버 버전의 swap을 만들어 넣고, 동시에 std::swap의 특수화 버전도 준비해야한다.

어떤 swap을 호출?

template<typename T>
void DoSomething(T &obj1, T &obj2){
  using std::swap;    //std::swap을 이 함수 안으로 끌고 들어온다.

  swap(obj1, obj2);
}

만약 swap을 사용할 것인데, 어떤 swap을 호출해야 될지 모를때는 먼저 “타입 T 전용 버전이 있다면 호출하고, 없다면 std의 일반형을 호출하도록 구현”한다. 그는 위의 예시 처럼 “using std::swap;”을 추가하면 된다.

단순히 기본 std의 일반형을 호출하고 싶다고 한정자를 붙여 “std::swap(obj1, obj2)”를 사용하면, 매커니즘에 영향이 가기 때문에 사용하지 마라.

요약

첫째, 표준에서 제공하는 swap이 클래스 및 클래스 템플릿에 대해 납득할 만한 효율을 보이면, 그냥 아무것도 하지 말고 쓰자.

둘째, 표준 swap의 효율이 기대한 만큼 충분하지 않다면 아래 과정을 따르자.

  1. 두 객체의 값을 빠르게 바꾸는 함수 swap을 public 멤버 함수로 만들자. (예외 X)
  2. 클래스 혹은 템플릿이 들어 있는 네임스페이스에 비멤버 swap을 만들어 넣고, 1번에서 만든 swap멤버 함수를 이 비멤버 함수가 호출하도록 한다.
  3. 클래스 템플릿이 아닌 클래스를 만들고 있다면, std::swap의 특수화 버전을 준비하고, 이 특수화 버전에서 swap 멤버 함수를 호출하자.

셋째, 사용자 입장에서 swap을 호출할 때, swap을 호출하는 함수가 std::swap을 볼 수 있도록 using 선언을 반드시 포함시키고, 네임스페이스 한정자를 붙이지 말자.

마지막, swap을 진짜 유용하게 사용하는 방법 중에서 클래스가 강력한 예외 안전성 보장을 제공하도록 도움을 주는 방법이 있기 때문에, 예외는 던지면 안된다.

잊지 말자

  • std::swap이 여러분의 타입에 대해 느리게 동작할 여지가 있다면, swap 멤버 함수를 제공하고, 예외를 던지지 않도록 처리하자.
  • 멤버 swap을 제공했으면, 이 멤버를 호출하는 비멤버 swap도 제공하고, 클래스(템플릿이 아닌)에 대해서는 std::swap도 특수화해주자.
  • 사용자 입장에서 swap을 호출할 때는, std::swap에 대한 using 선언을 넣어 준 후에 네임스페이스 한정 없이 swap을 호출하자.
  • 사용자 정의 타입에 대한 std 템플릿을 완전 특수화하는 것은 가능하다. 하지만 std에 어떤 것이라도 새로 추가하려고는 하지말자.
