항목 1 : c++를 언어들의 연합체로 바라보자.

c++은 “C, 객체지향의 C++, 템플릿의 C++, STL”과 같이 4가지 언어들의 기능을 모두 포함하고 있는데, 이 언어 중 어떤 기능을 사용하냐에 따라 프로그램의 규칙이 달라진다.

만약 예를들어 “값 전달”과 “참조 전달” 중 한가지 전달방법을 선택할때 언어에 따라 효과적인 전달방법이 있으며 아래와 같다.

기능 전달 이유
C 값 전달 형식적인 이유
객체지향의 C++ 참조 전달 생성자/소멸자 개념이 생겼기에
템플릿의 C++ 참조 전달 객체 타입조차 알 수 없기에
STL 값 전달 C의 포인터를 본떠 만든 것이기에

잊지 말자

  • C++은 경우에 따라 효과적인 프로그래밍 규칙이 달라진다. 이때 경우는 위에서 설명하는 4가지 언어를 의미한다.

항목 2 : #define을 쓰려거든 const, enum, inline을 떠올리자.

매크로

  #define A 1.6

매크로(선행처리자)로 위에서 선언된 A는 “기호식 이름”처럼 보이지만, 컴파일러에게 넘어가기 전에 선행처리자가 상수로 변환하기 때문에 A라는 이름은 “기호 테이블”에 들어가지 않는다.

※ 문제점 :

이는 에러발생 시, A는 “기호식 이름”으로 표시되지 않고 “상수”로 표시되기 때문에 혼란을 야기한다. 특히 다른 개발자가 매크로를 선언했을때는 오류를 찾을 수가 없다. 또한 선행처리자가 A를 찾아 모두 상수로 바꾸기 때문에 메모리 사용측면에서 비효율적이다.

※ 해결방법 :

  const double A = 1.6;

이를 해결하기 위해서는 매크로 대신 “상수”를 사용하는 것이다. 이는 기호 테이블에 들어가기 때문에 컴파일러에 표시되며, 여러번 쓰여도 사본은 1개만 생성되기 때문에 메모리 점유도 효율적이다.

※ 주의사항 :

상수 포인터를 정의하는 경우

  const char* const A = "~~";   //구시대적
  const string A("~~");
  • 위와 같이 포인터와 포인터가 가르키는 대상까지 const로 선언해야 되며, C언어의 구시대적 char*보다는 string 방식을 쓰는 것을 추천한다.

클래스 멤버로 상수를 정의하는 경우

이는 즉 클래스 상수를 공유하는 “정적(Static)”멤버를 만들때를 의미하며, 이때 해결 방법은 2가지가 존재한다.

※ 1번 방안

  //header
  class A{
  private:
    static const int tmp = 5;   //정의가 아닌 선언
    int score[tmp];
  };
  //cpp
  const int A::tmp;             //클래스 상수의 주소를 요구할때

이때 tmp는 선언시 초기화되며, 클래스 상수의 주소를 요구할 시, cpp에 정의해준다.

하지만 구 컴파일러의 경우 선언시점에 초기값 선언이 옳지 않다고 판단하게 될 수 있는데, 이는 아래와 같이 초기값을 정의 시점으로 옮겨서 해결한다.

  //header
  class A{
  private:
    static const int tmp;           //정의
  };
  //cpp
  const int A::tmp = 5;             //선언

※ 2번 방안

  //header
  class A{
  private:
    enum {tmp = 5;}
    int score[tmp];
  };

이는 만약 클래스 컴파일 도중 클래스의 상수값을 요구할때 나열자(enum)을 사용하여 해결한다.

이 나열자를 사용하면 수많은 장점이 존재하는데, 첫 번째로 선언한 정수 상수를 다른 개발자가 주소를 얻거나 참조하려할때 자물쇠역할을한다. 이를 통해서 접근을 제한한다.

두 번째로 1번 방안에서 const는 컴파일러의 오류로 혹시 모르는 메모리가 할당될 가능성이 있지만 나열자는 그런 가능성이 존재하지 않는다.

세 번재로 많은 개발자들이 이 방식을 사용하고 있으며, 이는 템플릿 메타프로그래밍의 핵심이다.

때문에 enum을 사용하는 것을 추천한다.

매크로 함수

  //매크로
  #define A(a,b) f((a) > (b) ? (a) : (b)

  //오류
  A(++a,b);

위와 같이 매크로 함수가 존재할때, ++a는 b와의 관계에 따라 2번 증가하거나 1번 증가하는 괴현상이 발생한다.

※ 해결방법 :

  template <typename T>
  inline void A(const T &a, const T &b){
    f(a > b ? a : b)
  }

위와같이 인라인함수에 대한 템플릿으로 구현한다. 이는 진짜 함수이기 때문에 유효범위와 접근규칙을 준수한다.

잊지 말자

  • 단순 상수는 #define이 아닌 “const 객체”,”enum”을 우선 사용하고, 매크로 함수는 “인라인 함수”를 사용한다.

항목 3 : 낌새만 보이면 const를 들이대 보자!

const의 대표적 예시

  1. 클래스 밖에서는 전역/네임스페이스 유효범위의 상수를 선언
  2. 파일, 함수, 블록 유효범위에서 static으로 선언한 객체에 사용 가능
  3. 클래스 내부에서는 정적/비정적 데이터 멤버 모두 상수 선언이 가능
  4. 포인터 자체 or 포인터가 가르키는 데이터를 상수로 지정 가능 
     const char * p;         //상수 데이터 
 char * const p;         //상수 포인터
  5. STL의 반복자(iterator) 사용시
    • 반복자는 자신이 가리키는 대상이 아닌 것을 가르키는 것도 가능하고, 대상 자체에 대해서도 변경이 가능하다.
    • 하지만 변경을 불가능하게 하기 위해서 “const_iterator”를 사용한다. 
       //이전
 const vector<int>::iterator it = vec.begin();
 *it = 10;     // 가능
 it++;         // it는 상수이기에 불가능

       //이후
 vector<int>::const_iterator it = vec.begin();
 *it = 10;     // *it는 상수이기에 불가능
 it++;         // 가능
  6. 함수 선언
    • 가장 강력한 용도로 사용되며, 반환 값/매개변수/멤버 함수 앞에 사용한다. 심지어는 함수 전체에도 선언하여 사용한다.
    • 이를 통해 기본 제공 타입과의 쓸데없는 비호환성을 파하는 것과 같은 에러가 감소하는데 예시는 아래와 같다.
     class A {};
 const A operator *(const A &a, const A &b);

 //Error
 A a,b,c;
 (a * b) = c;
 if(a * b = c);

멤버 함수가 상수 함수일때의 의미

개념 의미
비트 수준 상수성 (물리적) 정적 멤버를 제외한 어떤 데이터 멤버도 건들이지 않으며, 즉 객체를 구성하는 비트를 바꾸지 않는다.
논리적 상수성 비트 수준 상수성을 보완하는 개념으로 일부 몇 비트는 바꿀 수 있되, 사용자가 인지하지 못한다면 상관없다.

비트 수준 상수성

  • ‘제대로 const’로 동작하지 않는데도 검사를 통과하는 경우가 발생한다.
  • 아래와 같은 경우로 부적절하지만 비트수준 상수성이 있어서 허용된 경우이다. 이는 pText의 내용을 수정할 수 있는 큰 문제를 야기할 수 있다.
```c++
char& operator[](size_t t) const {
  return pText[t];                  
}

//ERROR
const A a("Hello");
char *pc = &a[0];
*pc = 'J';            //"Jello"로 값이 변한다.   
```

논리적 상수성

  • 위의 문제를 보오나하기 위한 개념으로, 일부 몇 비트는 바꿀 수 있다.
  • 아래와 같이 const로 선언된 상수 함수(length())가 있다고 가정할때, 이는 textLenght, lengthIsValid라는 변수에 대해서 값을 대입하고 있는 오류를 발생시킨다.
```cpp
private:
  char *pText;
  size_t textlength;
  bool lengthIsValid;
public:
  size_t length() const;

//length()
if(!lengthIsValid){
  textLength = strlen(pText);   //ERROR
  lengthIsValid = true;         //ERROR
}
```
  • 이를 해결 하기 위해서는 단순히 mutable을 사용하여 해결하면 되는데, 이는 비정적 데이터 멤버를 비트수준 상수성의 족쇄에서 풀어준다. 
    private:
  char *pText;
  mutable size_t textlength;    //해결
  mutable bool lengthIsValid;   //해결

상수/비상수 멤버 함수에서 코드 중복 배제하기

c++은 오버로딩을 지원하기 떄문에 만약 동일한 코드 내용인데 각각 상수와 비상수로 선언되어 있다면 이 코드 중복을 배제하기 위한 방안이 존재한다.

캐스팅이 필요하긴 하지만 안전성을 유지하고, 코드 중복을 피하는 방법은 operator[]가 상수 버전을 호출하도록 구현하는 것이다.

//상수 함수
const char& operator[](size_t postion) const { return text[postion]; }

//비상수 함수
char& operator[](size_t postion) {
  // op[]의 반환 타입에 캐스팅하여 const 제거
  return const_cast<char &>(                
    // *this의 타입에 const를 붙이고, op[]의 상수 버전을 호출
    static_cast<const A&>(*this)[postion]
  );
}

잊지 말자

  • const를 붙여 선언하면 컴파일러가 에러를 잡아내는데 도움을 주며, 굉장히 많은 곳에 붙을 수 있다.
  • 컴파일러 쪽에서는 비트수준 상수성을 지켜야 하지만, 개발자는 논리적인 상수성을 사용하여 프로그래밍해야한다.
  • 상수/비상수 멤버 함수가 기능적으로 똑같게 구현되어 있는 경우에는 코드 중복을 피하기 위해서 비상수 버전이 상수 버전을 호출하도록 한다.

항목 4 : 객체를 사용하기 전에는 반드시 초기화하자!

초기화를 하지 않으면 무작위로 비트 값을 읽어들이기 때문에 많은 문제가 발생한다.

비멤버 객체의 경우 직접 초기화를 진행하지만 멤버 객체의 경우 생성자를 통해 초기화를 진행한다.

//대입
A(a,b,c){
  q = a;
  w = b;
  e = c;
}

//초기화 (멤버 초기화 리스트)
A(a,b,c) :q(a),w(b),e(c) {}

위 코드의 경우 각각 대입과 초기화를 의미하는데, 대입은 기본 생성자 -> 복사 대입 연산자의 순서로 총 2번 이루어지지만, 초기화는 복사 생성자 1번만 이루어진다. 그렇기 때문에 초기화가 더욱 효과적이다.

// 기본 생성자로 구현된 멤버 초기화 리스트
A() : q(), w(), e() {}

기본 생성자로만 구현하고 싶더라도 멤버 초기화 리스트를 사용하는데, 이는 만약 어떤 멤버가 초기화 리스트에서 제외된 경우에는 기본 제공 타입이 되기 때문에 초기화가 되지 않을 수도 있다.

또한 상수, 참조자가 데이터 멤버인 경우에도 사용하는데, 이 데이터 멤버에는 대입 자체가 불가능하기 때문이다.

그렇기에 기본적으로 멤버 초기화 리스트를 사용하는 것이 좋다.

너무 코드가 지겹고 더럽게 느껴진다면, 대입으로 초기화가 가능한 데이터 멤버들을 리스트에서 빼내어 별도의 함수로 옮겨 이를 생성자에서 호출하는 방법이 있다. 이는 초기값을 파일이나 데이터베이스에서 읽는 경우 유용하다.

비지역 정적 객체의 초기화

개념 의미
비지역 정적 객체 전역 객체, 클래스/함수 내, 파일 유효범위 밖의 static 객체,네임스페이스 유효범위 객체
개별 번역 단위 컴파일을 통해 하나의 목적 파일을 만드는 바탕이 되는 소스 코드

비지역 정적 객체의 초기화 순서는 개별 번역 단위에서 지정되는데 이는 초기화 시 큰 문제를 야기할 수 있다.

예를 들어 2개의 컴파일된 소스 파일에 비지역 정적 객체가 1개 이상 존재하고, 한쪽에서 초기화를 진행할 때 다른쪽의 비지역 정적 객체를 사용하는데 초기화가 진행되어 있지 않다면 컴파일의 에러가 발생한다.

//1번 소스파일
class FileSystem { ~~ }
extern FileSystem tfs;    //비지역 정적 객체

//2번 소스파일
class Directory { 
  Directory() { a = tfs; }    //ERROR
}

이 문제를 해결하기 위해서는 비지역 정적 객체를 하나씩 맡는 함수를 준비하고, 이 안에 각 객체를 넣어 사용자 측면에서 비지역 정적 객체를 직접 참조하지 않고, 함수를 호출하는 것을 대신한다.

이때 비지역 정적 객체는 정적 객체로 바뀌며, 해당 객체 생성/소멸을 불가능하게 하여 비용이 안생기도록 처리한다.

//1번 소스파일
class FileSystem { ~~ }
FileSystem tfs(){
  static FileSystem fs;
  return fs;
}

//2번 소스파일
class Directory { 
  Directory() { a = tfs; }    
}

Directory &tempDir(){
  static Directory td;
  return td;
}

즉, 정적 객체 자체를 직접 사용하지 않고 그 객체에 대한 참조라를 반환하는 함수를 사용한다.

하지만 이는 내부적으로 정적 객체를 사용하기 때문에 다중스레드 환경에서는 동작에 문제가 생기는데, 이는 다중스레드 돌입전, 참조자 반환 함수를 손으로 호출하면 경쟁상황이 생기지 않는다.

잊지 말자

  • 기본제공 타입의 객체는 직접 손으로 초기화한다.
  • 생성자에서는 대입문을 사용하지 않고, “멤버 초기화 리스트”를 사용하고, 각 데이터 멤버가 선언된 순서와 똑같이 나열하자.
  • 여러 번역 단위에 있는 비지역 정적 객체들의 초기화 순서 문제는 피해야하기에 비지역 정적 객체를 지역 정적 객체로 바꾸자
