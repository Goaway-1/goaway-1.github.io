동시성 문제

동기

Dec 22, 2023 About 5 mins

#UE5#UnrealEngine

개요

개발하다 보면, 엔진이나 프로그래밍 차원에서 발생할 수 있는 문제들이 존재합니다. 이번에는 프로그래밍에서의 ‘동기’라는 개념에 대해서 알아보고, 언리얼 스레드 환경에서 발생할 수 있는 동시성 문제에 대해서 설명해 보겠습니다.

프로그래밍에서의 동기 & 비동기

‘서버와 로컬관계’에서의 동기는 데이터를 동일하게 맞추는 것을 동기화라고 부릅니다. 하지만 ‘프로그래밍’에서의 동기란 대기하는 작업들에 대해서 순차적으로 처리되는 것을 의미하죠. 반대로 비동기는 순차적으로 발생하는 것이 아닌, 스레드를 사용하여 여러 가지 일이 동시에 처리되는 것을 의미합니다.

언리얼 스레드

언리얼 엔진은 기본적으로 메인/게임 스레드와 여러 개의 작업 스레드를 사용합니다. 이전 “언리얼 프로파일링”이라는 포스트에서도 작성한 적이 있는데요. 언리얼 엔진은 작업 스레드를 사용하여 다양한 작업을 병렬로 처리함으로써 성능을 향상시키고 멀티코어 시스템에서 효과적으로 작동할 수 있도록 설계되어 있습니다. 하지만 스레드의 정확한 개수는 엔진의 버전 및 구성, 사용 중인 플랫폼에 따라 다를 수 있고, 엔진의 설정 파일에서 스레드 관련 설정을 변경할 수 있으므로, 프로젝트에 따라 다르게 조정될 수 있습니다.

  1. 메인/게임 스레드 (Main/Game Thread): 게임 로직, 렌더링 및 사용자 입력 처리 등의 주된 작업을 담당하는 스레드입니다.
  2. 렌더 스레드 (Render Thread): 렌더링 작업을 처리하는 스레드입니다. 그래픽스 API 호출 및 렌더링 명령을 실행합니다.
  3. 비동기 로딩 스레드 (Async Loading Thread): 언리얼 엔진은 비동기 로딩을 지원하기 위해 별도의 스레드를 사용합니다. 이 스레드는 애셋을 비동기적으로 로드하고 언로딩하는 작업을 처리합니다.
  4. 특정 작업을 위한 추가 스레드: 언리얼 엔진은 특정 작업을 병렬로 처리하기 위해 추가적인 스레드를 생성할 수 있습니다. 예를 들어, 빌드 라이트맵이나 네트워크 통신을 담당하는 스레드 등이 여기에 속합니다.

동시성 문제

언리얼 엔진의 스레드는 이전 설명했던 것과 같이 여러 개가 있고, 사용자가 직접 스레드를 추가할 수도 있습니다. 즉, 다양한 작업을 병렬로 처리하는 비동기 방식으로 동작한다는 것이죠. 이렇게 되면 어떤 문제가 발생할 수 있을까요? 바로 “동시성”이라는 골치 아픈 상황에 직면할 수 있습니다.

동시성이란 쉽게 말해서 동일한 자원에 대해 여러 스레드가 동시에 접근하면서 발생하는 문제입니다. 물론 무조건 동시에 접근한다고 해서 문제가 발생하는 것이 아닌, 값은 누군가가 변경할 때 발생하게 됩니다. 이렇게 문제가 발생해서 예상했던 결과가 달라질 수 있는 상황을 경쟁 조건(Race Condition)이라고 합니다.

이는 언리얼 엔진의 게임 스레드에서 발생할 수 있을 것 같은데요. 비록 하나의 게임 스레드에서 모든 로직이 실행되기는 하지만, 여러 이벤트와 함수 호출이 동시에 발생할 수 있고, 그중 일부가 상태를 변경하거나 공유 데이터에 접근할 때 동시성 문제가 발생할 수 있습니다. 동시성 문제의 주요 원인은 다음과 같습니다:

  1. 이벤트 및 비동기 호출: 여러 이벤트 또는 비동기 호출이 동시에 발생할 때, 이들이 동일한 데이터를 변경하거나 접근할 경우 문제가 발생할 수 있습니다.
  2. 다중 스레드 접근: 게임 스레드 외에도 다른 스레드에서 공유 데이터에 접근하면 문제가 발생할 수 있습니다. 예를 들어, 비동기 로딩 스레드가 로딩된 애셋과 관련된 데이터를 변경하고 게임 스레드에서 이를 사용하려 할 때 문제가 발생할 수 있습니다.

해결 방법

동시성 문제를 해결하기 위해 언리얼 엔진은 다양한 기능을 제공합니다. FCriticalSection 클래스를 사용하여 크리티컬 섹션을 정의하고, 이를 이용해 특정 부분에 대한 상호 배제를 구현할 수 있습니다. 또한, 액터의 상태 변경 등에 대한 작업을 FQueuedThreadPool을 사용하여 백그라운드 스레드에서 처리할 수 있습니다.

이번에는 FCriticalSection 클래스를 사용하는 방법에 대해서만 설명해 보겠습니다. 이는 스레드에서 안전하게 액세스할 수 있도록 필요한 작업을 완료할 때까지 하나의 스레드 잠금으로 처리하는 방식을 취합니다. 잠금/해제 로직을 처리하는 방법에는 스레드를 직접 잠그거나 RAII 기술을 사용하는 2가지 방법이 존재합니다.

수동

스레드를 직접 잠구거나 해제하는 방식인데, 이는 데드락이 발생할 수 있습니다.

class AnyClass
{
	FString StringToEdit;

	// Marked as mutable as it's often warranted by design
	mutable FCriticalSection DataGuard;

public:
	void SetString(const FString& NewString)
	{
		DataGuard.Lock();
		StringToEdit = NewString;
		DataGuard.Unlock();
	}
};

RAII

이는 특정 범위 내에서 자동으로 처리해주기 때문에 훨씬 안전하게 사용할 수 있습니다.

class AnyClass
{
	FString StringToEdit;

	// Marked as mutable as it's often warranted by design
	mutable FCriticalSection DataGuard;

public:
    void SetString(const FString& NewString)
	{
		FScopeLock Lock(&DataGuard);
		StringToEdit = NewString;
	}
};

이외에도 C++ 11에 추가된 std::atomic 라는 템플릿이 존재하는데, 이는 메모리 순서화 기능이 포함되어 있어 스레드 안정성을 보장합니다. 데이터를 안전하게 공유하기 위한 작업에서 사용하며, 동시 접근 시 발생할 수 있는 경합 조건을 방지할 수 있죠. 다만 언리얼 엔진 5부터는 사용하지 말라고 하네요. ㅠㅠ

추천 해결 방법

위와 같은 방법으로 해결할 수 있지만, 가장 좋은 방법은 애초에 상호 배제를 최소화하고 데이터 공유를 피하는 등의 방법을 사용하여 동시성 문제를 최소화하는 것이 최고입니다….!

저도 실제로 게임을 개발하며, 발생했던 이슈가 있었는데요. 캐릭터의 공격 상태나 아이템에 의한 효과 등 여러 개의 스탯에 따라 이동속도에 영향을 받도록 구현한 적이 있습니다. 하나의 스탯이 변경되면 그 즉시 이동속도가 변경되도록 구현했었는데, 이때 2개 이상의 스탯이 동시에 변경되어 각각 이동속도를 변경하게 된다면, 이동속도를 수정하기 위해서 동시에 접근하는 문제가 발생해 값이 꼬이는 문제가 발생할 겁니다.

이를 해결하기 위해 멘토님이 말씀해 주신 가장 간단한 방법은 즉시 적용하지 않고, “스레드, 타이머, 틱”을 사용하여 유연하게 각 스탯의 값 변경을 체크하고 이동속도를 변경하는 방식이었습니다. 이렇게 하면, 동시에 접근하는 문제가 발생하지 않겠죠?

마무리

게임 개발하며 발생할 수 있는 동시성 문제에 대해서 작성해 보았는데요. 제가 알려드린 방법은 극히 일부이고, 상황에 따른 문제해결 방법은 정말 수 없이 많을 겁니다. 항상 정형화된 방법은 없는 거니까요. 그럼 오늘도 즐거운 개발되세요!
