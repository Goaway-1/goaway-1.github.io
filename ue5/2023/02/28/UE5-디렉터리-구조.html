디렉터리 구조

언리얼 엔진이 설치된 엔진 디렉터리와 프로젝트 디렉터리로 구분되며, 두 디렉터리에는 ‘공통 디렉터리’가 포함되고, 추가로 엔진 디렉터리는 ‘루트와 엔진 전용 디렉터리’를 포함한다.

루트,엔진 전용 디렉터리

엔진 디렉터리에 포함된 내용들로, ‘루트, 엔진 전용’ 디렉터리의 파일들을 합친 것이다.

폴더 내용
Engine 엔진을 이루는 모든 소스 코드, 콘텐츠 등이 포함
Templates 새 프로젝트 생성시 사용할 수 있는 프로젝트 템플릿 모음
FeaturePacks, Samples 스타트 콘텐츠 모음 (추가 설치 필요)
GenerateProjectFiles.bat Visual Studio 에서 엔진과 게임 작업시 필요한 UE4 솔루션 및 프로젝트 파일 생성시 사용 (추가 설치 필요)
UE4Games.uprojectdirs 엔진이 하위 디렉터리의 프로젝트르 발견할 수 있도록 도와주는 헬퍼 파일 (추가 설치 필요)
Documentation
  • 엔진 문서 소스와 퍼블리시된 파일.
    - HTML : 퍼블리시된 HTML 문서 파일.
    - Source : 소스 마크다운 문서 파일.
Extras 부가 헬퍼, 유틸리티 파일. Visual Studio의 Custom Visualizer를 Eidtor에서 사용하기 위한 설정 값 저장
Plugins 엔진에 사용되는 플러그인
Programs
  • UE4 루트 디렉터리에 저장된 프로젝트와 “UnrealFrontend, UnrealHeaderTool” 등의 기타 언리얼 프로그램용 환경설정 파일과 로그 파일 존재.
    - 빌드툴과 헤더툴이 존재하며, 빌딩의 자동화를 위한 도구. (참조)
Shaders 엔진에 대한 셰이더 소스 (.usf) 파일

※ 본인은 “GenerateProjectFiles.bat, UE4Games.uprojectdirs”파일을 찾아볼 수 없었는데, 언리얼엔진 공식 깃허브에 가면 확인할 수 있다.

공통 디렉터리

폴더 내용
Binaries 컴파일 도중 생성되는 실행 파일이나 기타 파일이 포함
Build 엔진이나 게임을 빌드하는 데 필요한 파일, 플랫폼별 빌드를 만드는 데 필요한 파일 포함
Config
  • 엔진 행위를 제어하는 값 설정용 환경설정 파일. 게임 프로젝트 Config 파일에 설정된 값은 Engine\Config 디렉터리에 설정된 값을 덮어씀.
    - 에디터, 게임, 하드웨어등 기본 설정값들이 어떻게 사용되는지 유추 가능.
    - 지워도 상관없지만, 설정해둔 값들이 모두 삭제되고, 지우고 나서 실행해도 다시 생성되지 않음
Content Engine 이나 게임에 대한 콘텐츠, 애셋 패키지와 맵 존재 (.uasset 타입)
DerivedDataCache 참조된 콘텐츠에 대해 로드시 생성된 파생 데이터 파일이 존재하지 않으면 로드 시간이 엄청나게 길어짐
Intermediate
  • 엔진이나 게임 빌드 도중 생성된 임시 파일. 게임 디렉터리에서 셰이더는 Intermediate 디렉터리에 저장.
    - 프로젝트 작업 시 임시로 저장되는 파일들이 보관되는 장소로, “자동 저장, 스크린샷, 빌드 파일, 백업 임시 파일, 로그, SaveGames”등 많은 데이터가 저장.
    - 언리얼 에디터 사용 시 작업의 효율성을 높이거나 복구작업, 문제 확인 등의 활용성이 높은 폴더. 다만 작업이 길어지고 패키징까지 하면 용량을 꽤 많이 차지.
Saved
  • 자동저장, 환경설정 (.ini), 로그 파일.
    - 추가적으로 Engine/Saved 디렉터리에는 크래시 로그, 하드웨어 정보, 스웜 옵션 및 데이터가 포함
Source
  • Engine 이나 게임은 물론 엔진 소스 코드, 툴, 게임플레이 클래스 등에 대한 모든 소스 파일.
    <blockquote> - Engine
    <blockquote>- Developer : 에디터와 엔진 둘 다에 쓰이는 파일.
    - Editor : 에디터에만 쓰이는 파일.
    - Programs : 엔진이나 에디터에 쓰이는 외부 툴.
    - Runtime : 엔진에만 쓰이는 파일. </blockquote>
    - Game
    <blockquote>- Classes : 모든 게임플레이 클래스 헤더 (.h) 파일.
    - Private : 모든 .cpp 파일과 게임플레이 클래스 구현 파일, 모듈 구현 파일.
    - Public : 모듈 헤더 파일.

재빌드 과정

본인은 프로젝트에 문제가 발생하거나 재빌드하고 싶다면, “Binaries, Build, Intermediate, DerivaedDataCache, Save, Visual Studio 파일(.sin)”와 같은 파일들을 모두 지우고, 새로운 Visual Studio 파일(.sin)을 생성하고 컴파일 후 재빌딩한다.

하지만 이때, ‘DerivaedDataCache나 Saved’폴더는 삭제에 있어 유의해야한다. 상관은 없지만 DerivaedDataCache같은 경우에는 콘텐츠에 대해 캐시 파일이 존재하기에 로드 시간이 길어질 수 있다.

Saved폴더 같은 경우에는 스크린샷이나 빌드 파일, 로그등등의 데이터가 축적되기 때문에 작업의 효율성을 높이거나 복구 작업, 문제 확인등에 활용하기 때문이다. 그래서 일부 개발자는 색상 정보를 저장하는 Saved/Config를 제외하고 삭제한다. 이는 상황에 맞게 사용하는 것을 추천한다.

협업시 관리 디렉터리

협업 상황에 있어 Git이나 언리얼에서 제공하는 SVN을 사용할때, 어떤 파일을 한정해서 관리해야 하는지에 대한 설명이다. 버전관리 툴로 디렉터리를 관리할 때는 빌드나 컴파일을 통해서 다시 생성되지 않는, 필수적인 파일들만 관리를 해야한다.

그렇기에 “Saved, Intermediate, Build, .vs파일, .sln파일”와 같은 디렉터리와 파일을 버전관리에 있어 제외해야한다. 이들은 빌드하면 다시 생성되는 정보들이기 때문이다. 이들을 제외한 “.uproject 파일, Binaried, Config, Contect, Source, Plugins”를 관리해주어야한다.
